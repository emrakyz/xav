{P}▌ {C}1.2  {Y}Why?
{P}━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  {P} {B}10bit {W}provides more levels per channel which greatly reduces banding
  {P} {W}Modern codecs compress {B}10bit {W}better
  {P} {W}Internal calcs have more precision; fine details survive better
  {P} {W}Less artifacts due to increased precision
  {P} {B}yuv420p10le {W}({B}10bit{W}) is supported by almost all hardware decoders
  {P} {W}Motion estimation, transforms, & rate control work with higher precision
  {P} {W}Better to micro-optimize, internally defaulting to {B}10bit
  {P} {B}8b {W}accumulates rounding error across every forward/inverse transform round trip
  {P} {W}Banding lives in dark, slow gradients (skies, fades, shadows), where lower range runs out
  {P} {W}HDR needs {B}10bit {W}anyway
  {P} {W}No benefit in encoding in low bit depth mode decisions or {B}yuv420p
  {P} {W}Encoder math runs {B}16bit {W}for high bit depth; feeding it {B}10bit {W}keeps whole
    {W}pipeline in its precise path instead of legacy {B}8bit {W}one
  {P} {W}If you have to use {B}8b, {R}do not use XAV

{P}▌ {C}1.7  {Y}Crop
{P}━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  {P} {W}Detects crop (consistent dark pixels, also called letter/pillar-boxing)
  {P} {W}Crop makes scene-detect, memory flow, filter, process, encode faster & raises efficiency
  {P} {W}It is instant & very robust
  {P} {C}Content-safe by construct: {W}One bright frame's one side exposing true edge overrides
    {W}every dark frame/sides; sampling spread exists to catch that frame
  {P} {C}Variance tolerance:{W} Analog & mosquito noise, not-quite-black bars are absorbed by variance check

{P}▌ {C}1.9  {Y}Scene Detect
{P}━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  {P} {W}Improves quality/efficiency
  {P} {W}Crop happens before SCD; cuts are computed on cropped frames
  {P} {W}Detailed & complex scene detection with rules:
      {C} {W}Detects based on encode heuristics (not human understanding of scenes)
      {C} {C}Min allowed length {W}can be anything: Allows refreshing references when needed
      {C} {C}Max allowed length {W}is {B}300 {W}frames ({B}12.5 seconds {W}for {B}23.976 FPS {W}and {B}5-sec {W}for {B}60 FPS{W})

          {P}· {W}Decoders may have problems decoding {B}300+
          {P}· {W}Seeking becomes harder with {B}300+
          {P}· {W}XAV tries to use RAM/VRAM very efficiently; longer GOPs will kill your memory
          {P}· {B}300-max: {W} Evidently absolute sweet spot
  {P} {W}End of SCD; redistributes scenes (called {C}enhanced-split{W}):
      {C} {C}Enhanced-split {W}is global (whole source), not greedy/local
      {C} {W}During SCD, frames get scores (such as calculated inter, intra costs)
        {W}Instead of cutting from boundaries, all scores are re-evaluated
        {W}to pick best possible combination of cuts
      {C} {W}a long static/slow shot that never triggers a content cut still gets split
        {W}at {B}300 {W}(or redistributed to be cut before) but it will never exceed {B}300
  {P} {C}Cuts can be back-to-back. {W}Because min length is unconstrained, hard cuts in fast
    {W}action (flashes, rapid intercuts) each get their own keyframe instead of being smeared into one GOP
  {P} {C}Cost model is decider. {W}A cut is placed when a fresh intra frame is cheaper than continuing to predict
    {W}frame intra cost vs inter cost; not pixel-difference thresholds
  {P} {W}Frame 0 (first frame) of each chunk is always a keyframe
  {P} {C}Handles gradual transitions. {W}Fades/dissolves/crossfades are caught by cost model
  {P} {W}If audio is encoded, all selected audio streams are encoded at same time along with SCD
    {W}This makes whole process faster. If scene file already present, audio encodes in end
  {P} {C}SCD defines parallel work units. {W}Scene boundaries are chunk boundaries, so this stage
    {W}directly sets what workers encode, not just where keyframes go
  {P} {C}Deterministic: {W}Same input {P} {W}identical scene file, every run/machine
  {P} {C}File is inspectable & editable: {W}0-indexed, one keyframe per line, last scene runs implicitly to EOF
    {W}You can hand-tweak a boundary & re-run
  {P} {W}It is a simple text file with keyframe locations each line:

{P}▌ {C}1.10  {Y}Muxing
{P}━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  {P} {W}XAV has {B}2 {W}containers: {B}mkv{W}/{B}webm
  {P} {W}It is hand-written EBML: No libmatroska/libebml/ffmpeg-libs: so every byte is controlled and
    {W}output is byte-deterministic
  {P} {C}Why custom: {W}Full byte-control, smallest overhead, zero dependency, more features, much faster
  {P} {B}MKV {W}is recommended mode for everything. It's a customized muxing which is:
      {C} {G}Fully compliant
      {C} {W}Has all additional features implemented to make it robust:
          {P}· {W}Streamable/seekable directly through download
          {P}· {W}Per-frame, per-level {C}CRC32 {W}error detection (corrupted frames can be detected)
          {P}· {W}Embedded metadata as muxer/encoder libraries, encoding params, date, etc
          {P}· {W}XAV stamps itself + versions so file is self-identifying
          {P}· {W}With all timing related information in-place (proper seeking, reverse-playback, etc)
  {P} {W}Audio/subtitle streams are muxed in with language/title tags & default/forced flags
  {P} {W}Colorimetry written at container level too; not just bitstream
  {P} {C}SAR/DAR {W}written so anamorphic content displays correctly without re-deriving from pixels
  {P} {C}Cues / SeekHead: {W}Explicit index so seeking is {B}O(1) {W}(constant time complexity), not a linear scan
  {P} {C}Minimal overhead: {W}No wasted lacing/padding; container bloat is measured against bitstream
  {P} {W}Timestamps derived from parsed information & derived from how XAV encodes videos deterministically
    {W}Exact CFR timing, no rounding drift across a long file
  {P} {W}Final assembly produces same container bytes regardless of chunk encode order
  {P} {W}Every player/parser/editor resolves timestamps identically, no exotic-tick edge cases
  {P} {C}Total Duration & number of frames written up front {W}(in Info): players know full length when
    {W}they open header. Scrubber, end-time, & remaining-time are correct before a single cluster is read,
    {W}not estimated by scanning to EOF
  {P} {C}Cluster-per-scene = every seek target is a keyframe {W}; clusters are aligned to scene/GOP boundaries
    {W}a seek can only ever land on a clean random-access point; never mid-GOP, never decoding from a
    {W}non-keyframe into garbage
  {P} {C}Has complete Cues, not simple ones. {W}Each {C}CuePoint {W}carries full set:
      {C} {C}CueTime{W}: exact timestamp of seek point
      {C} {C}CueClusterPosition{W}: Segment-relative byte offset of cluster (jump straight there, zero scan)
      {C} {C}CueRelativePosition{W}: Offset of block within cluster, so demuxer lands on exact frame,
        {W}not just cluster start
      {C} {C}CueDuration{W}: How long cue point covers
      {C} {C}CueBlockNumber / CueTrack{W}: Deliberate block + track identity
  {P} {W}Many muxers omit {C}CueRelativePosition {W}and {C}CueDuration{W}; without them a seek resolves to "somewhere in
    {W}this cluster" & player re-parses to find frame. XAV's cues resolve to exact block in one jump
  {P} {W}Inherits explicit per-frame {C}BlockDuration {W}(via {C}BlockGroup{W}). Every frame states its own duration instead of
    {W}being inferred from next frame's timestamp. Last frame of a file, or a frame before a gap still has a correct,
    non-mistakable duration
  {P} {C}ReferenceBlock {W}reference structure where p-frames declare what they reference; keyframes declare none
    {W}so a demuxer knows exact GOP dependency graph & can drop cleanly to nearest keyframe on seek
    {W}without speculative decoding
  {P} {C}Backward PrevSize chain {W}allows every non-first cluster records its predecessor's size to form
    {W}a doubly-linked cluster list. This is what makes smooth reverse playback & resync after a damaged
    {W}cluster possible; a parser that loses its place can walk backward to a known boundary instead of giving up
  {P} {C}Self-locating clusters {W}store their own Segment-relative Position; file can be
    {W}re-indexed from any point even if Cues element is lost or truncated
  {P} {W}File carries a unique segment identity;external tools can reference it reliably across renames/moves
  {P} {W}Secondly, {B}webm {W}can be used as an experimental minimal muxer which has almost nothing
      {C} {W}It does not aim for full compliance & it is barely playable
      {C} {W}It can be handy for {B}<1MB {W}challenges
      {C} {W}Will still work on Discord/Browsers/Apps. It just does not have advanced features
  {P} {W}A future projection to allow more flexibility for muxing, exists (adding attachments, etc)

{P}▌ {C}1.12  {Y}Target Quality
{P}━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  {P} {W}TQ is peak of encoding with a known drawback: {R}Time Spent
  {P} {W}Uses psychovisually improved video quality metrics that are perceptually relevant to us
    {W}compared to ancient metrics like {B}PSNR/SSIM/VMAF {W}or variants
  {P} {C}Psychovisual Metric {W}is a video quality metric that models human neurosensory features and
    {W}display capabilities, viewing room characteristics & viewing distance; a metric that aligns with
    {W}human subjective evaluations as to what image/video quality means
  {P} {C}SSIMULACRA2 {W}& {C}Butteraugli {W}do this through construction; however {C}CVVDP {W}is more flexible
    {W}in how you define viewer & display characteristics
  {P} {W}TQ defines a target {Y}RANGE {W}(not a single value):
      {C} {W}"I want every scene to be between {B}N-N {W}objective quality" (where {B}N {W}is a quality number/score)
      {C} {W}"It can be higher than THIS quality & lower than THAT quality" (you accept anything between)
  {P} {W}Scenes are individually analyzed & improved until they are exactly in your target range
  {P} {W}Metrics used are {C}CVVDP {W}or {C}Butteraugli 5pNorm {W}or {C}SSIMULACRA2
  {P} {W}CRF search is not done naively. There are many intelligent mechanisms providing faster convergence,
    {W}and many early exist capabilities
  {P} {W}Scene is encoded {P} {W}then quality-tested {P} {W}If in range; {G}saved{W}, if not; gets re-encoded
  {P} {W}This process continues until requested range is reached for that scene
  {P} {W}TQ encoding takes {B}2.0x {W}to {B}3.5x {W}total encoding time (depending on your config)
  {P} {W}Main usecases:
      {C} {C}Testing purposes: {W}You can compare params and/or encoders on certain scenes or full content by
        {W}normalizing on an objective quality level. Based on output bitrate; you can determine winners on your
        {W}quality range/target
      {C} {C}Automation: {W}You don't have to know what CRF to use; system will pick correct one each scene
      {C} {C}Normalizing/Consistency: {W}Plain CRF is naive. Every scene will have different quality
        {W}creating inconsistencies. TQ improves consistency greatly
        {W}Internal encoder metrics are too simple, that no known encoder can create consistent/robust quality
        {W}throughout a whole movie without relying on external pipelines
      {C} {W}You don't want to visually analyze every content you encode; OR you don't trust your eyes as sole
        {W}determiner. A metric score is generalizable
      {C} {C}To create advertisable encodes: {W}You can show that your video has a {C}CVVDP {W}score of {B}9.9
        {W}which will make your video objectively relevant even before viewing it
      {C} {C}To save bitrate: {W}Some people want very high quality but using a safe-enough CRF will bloat many
        {W}scenes' bitrate when those scenes don't need that high of a bitrate. TQ will clamp them
        {W}where it will also boost scenes that still sit in lower quality side even with that safe-enough CRF
  {P} {W}Plain CRF fixes effort & lets quality float. TQ fixes quality & lets bitrate float

{P}━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

{P}▌ {C}2.8  {P}┃ {C}--hwdec      {W}Use GPU decode
{P}━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  {P} {W}Enables GPU Decoding
  {P} {W}Valid when input files have codecs: {B}AVC (H.264){W}, {B}HEVC (H.265){W}, {B}VC-1{W}, {B}VP9 {W}& {B}AV1
  {P} {B}VC-1 {W}is only supported by Nvidia GPUs through CUDA. You need:
      {C} {W}Nvidia GPU
      {C} {W}Build with TQ (because we link CUDA for TQ)
  {P} {W}However codec list is GPU + driver dependent (some won't support AV1 for example)
  {P} {R}No CPU fallback. {W}If {C}--hwdec {W}is used & it fails, run errors out
  {P} {W}HWDEC is preferable because GPU generally decodes much faster, especially when:
      {C} {W}Input is {B}VC-1 {W}(on CPU, it is single threaded due to format constraints)
      {C} {W}Input is a complex HEVC (many relatively modern GPUs decode HEVC quite efficiently)
  {P} {W}Other times it can still be beneficial because CPU gets completely free from decoding
    {W}The whole CPU resources are used for encoding instead
  {P} {W}However real benefit is hardware parallelism, not just freeing CPU
    {W}Decode runs on GPU while CPU encodes; you use a hardware that otherwise sleeps
  {P} {W}GPU decode is also much more power efficient
  {P} {W}Can not be combined with pipe mode because decoding happens on
    {W}sender tool (command you pipe into xav)

{P}▌ {C}2.10  {C}-a {P}┃ {C}--audio      {W}Opus Enc: {Y}-a {G}"{R}<{G}auto{P}┃{G}norm{P}┃{G}bitrate{R}> {R}<{G}all{P}┃{G}stream_ids{R}>{G}"
{P}━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  {P} {W}Two different space separated segments:

          {P}· {W}Auto calculated bitrate
          {P}· {W}OR your own bitrate
          {P}· {W}OR want to use stereo downmixing + loudness normalization where bitrate is AUTO
            {W}However norm allows more flexibility which will be explained later
      {C} {C}Second segment {W}is audio streams you want to encode to OPUS (the others are REMOVED):
          {P}· {W}If {G}all {W}is selected then all audio streams will be encoded to OPUS
          {P}· {W}If a number is selected (check input audio numbers on mediainfo/ffprobe)
            {W}numbered audio streams are encoded & others are discarded
          {P}· {W}The IDs can be comma separated to encode more audios: {B}1,3,5
  {P} {W}Put these two segments in double quotes
  {P} {W}Auto bitrate calc is channel based. Results of formula used:
      {P} {B} 76 {W}for mono {B}(1.0)
      {P} {B}128 {W}for stereo {B}(2.0)
      {P} {B}132 {W}for {B}2.1
      {P} {B}177 {W}for {B}3.1
      {P} {B}219 {W}for {B}4.1
      {P} {B}258 {W}for {B}5.1
      {P} {B}295 {W}for {B}6.1
      {P} {B}331 {W}for {B}7.1

  {Y}▍ How Do Stereo Downmixing & Loudness Normalization Work?

  {P} {W}Stereo downmixing is based on {B}AC4 {W}standards: {B}ETSI TS 103 190-1{W}, Section {B}6.2.17
  {P} {B}5.1 (6ch){W}: {B}Lt = L + 0.707*C + 0.707*Ls{W}, {B}Rt = R + 0.707*C + 0.707*Rs {W}; surrounds at {B}−3 dB{W}, LFE discarded
  {P} {B}6.1 (7ch){W}: back-center folded in at {B}0.5 (−6 dB) {W}to each side; surrounds {B}−3 dB
  {P} {W}For {B}7.1{W}, it's cascaded; side surrounds are pre-scaled by {B}0.707{W}, then {B}5.1->2.0 {W}stage applies another {B}0.707
      {C} {W}computed as a single fused coefficient matrix per sample; not two sequential passes
        {W}(no intermediate buffer or double rounding)
  {P} {W}There is also channel reordering that happens before all of these due to OPUS format
    {W}constraints (Vorbis layout)
  {P} {W}Mono/Stereo sources are passed through untouched even if you use norm option

  {Y}▍ Then Loudness Normalization is Applied:

  {P} {W}XAV uses {B}EBU R128 {W}/ {B}ITU-R BS.1770 {W}model
  {P} {W}Enabled via {C}norm {W}or {C}norm(I,TP,LRA[,BITRATE])
  {P} {W}Plain {C}norm {W}option uses defaults
  {P} {W}Three values are standard R128 triplet, plus Opus bitrate
  {P} {C}Pass 1: {W}Track is downmixed to stereo first, then measured with EBU R128
    {W}(integrated loudness + true-peak + loudness range). Normalizes what you'll actually hear,
    {W}not original surround
  {P} {C}Pass 2: {W}One constant gain is computed & applied to every sample, then peaks are clamped
    {W}to true-peak ceiling
  {P} {C}Internal dynamics are preserved. {W}LRA is a coarse global pull down, not a compressor; and
    {W}true-peak stage is a brickwall clamp, not a lookahead limiter

  {Y}▍ Values & Their Defaults

  {P} {C}I {W}(default is {B}-16.0{W}) {P}: {W}Target integrated (overall) loudness whole track is shifted to
    {W}More negative = quieter. Uses {B}LUFS {W}unit
  {P} {C}TP {W}(default is {B}-1.5{W}) {P}: {W}True-peak ceiling. Post-gain samples are clamped here; leaves headroom below {B}0 dBFS
    {W}Uses {B}dBTP {W}unit
  {P} {C}LRA {W}(default is {B}16.0{W}) {P}: {W}Loudness-range target. If source exceeds it, overall gain is pulled down toward it
    {W}narrower sources are left alone. Uses {B}LU {W}unit
  {P} {C}BITRATE {P}: {W}Output Opus bitrate for normalized stereo track. Uses {B}kb/s {W}unit

  {Y}▍ Why These Defaults

  {P} {C}I = -16 LUFS {P}: {W}Middle ground value used by Apple Music. Spotify/YouTube uses {B}-14
      {C} {W}Within Audio Engineering Society recommended range ({B}-16..-20{W})
  {P} {C}TP = −1.5 dBTP {P}: {W}Leaves {B}1.5 dB {W}of headroom for inter-sample overshoot & lossy-codec ringing
    {W}Value is smallest value that could reliably prevented clipping in default scenario. Not randomly picked
  {P} {C}LRA = 16 LU {P}: {W}Intentionally permissive. {B}16 {W}preserves natural film/music dynamics & only engages for pathologically
    {W}wide sources. Goal: "don't squash too much". Otherwise it would feel too flat
  {P} {C}BITRATE = 128 kbps {P}: {W}Perceptually transparent-grade Opus for stereo
  {P} {C}Output always stereo: {W}Normalization runs on downmixed 2-channels & encodes stereo Opus, regardless of channel count
  {P} {C}Higher level overview: {W}In summary Normalization helps with some of issues we can have:
      {C} {W}We can use much lower bitrates (instead of {B}330{W}, we can use {B}128 {W}or below)
      {C} {W}Opus already doesn't support some exotic channel types or audio features. However it is golden for stereo
      {C} {W}Many people don't have proper setups to play surround audio properly:
          {P}· {W}Extremely expensive
          {P}· {W}No space to place audio systems
          {P}· {W}No environment to play sound loud enough that it would matter
      {C} {W}Surround audio has a range that is too wide; & it doesn't play well on standard systems or headphones
      {C} {W}Especially dialogue (most important aspect in a movie) suffers
      {C} {W}Default values preserve original sound while suppressing wideness a little while improving dialogue loudness
      {C} {W}Generally downmixing + normalization creates a "louder" feeling which people commonly like
